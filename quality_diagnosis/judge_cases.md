## Case 1

### Question

결제는 완료되었는데 주문 내역에 보이지 않습니다.

### Candidates

- **S0**: 결제는 완료되었는데 주문 내역에 보이지 않습니다. | 근거: [VOC1] CUST090 결제는 완료되었는데 주문 내역에 보이지 않습니다.
- **S1**: 결제 완료 주문 내역 미노출 주문 누락 | 근거: [VOC2] CUST103 결제 완료 주문 내역 미노출 주문 누락
- **S2**: CUST090 결제는 완료되었는데 주문 내역에 보이지 않습니다. | 근거: [VOC1] CUST090 결제는 완료되었는데 주문 내역에 보이지 않습니다.

### Policy

#### 정책 개선안: 결제-주문 내역 동기화 오류 개선

##### 우선순위 1 (긴급) - 결제/주문 시스템 정합성 확보

**담당 조직**: 결제시스템팀, 주문관리팀

**무엇을**: 결제 완료 트랜잭션과 주문 내역 DB 간 실시간 동기화 로직 점검 및 장애 원인 규명

**언제까지**: 2주 이내 (긴급 이슈로 분류)

**어떻게**:
- 결제 완료 시점과 주문 내역 생성 시점 간 딜레이 또는 유실 구간 로그 전수 조사
- 결제 성공 후 주문 내역 생성까지 타임아웃 및 재시도(Retry) 로직 신규 적용
- 결제-주문 데이터 불일치 건에 대한 자동 알림 모니터링 시스템 구축 (불일치 발생 시 5분 내 담당자 알림)

---

##### 우선순위 2 (높음) - 고객 대응 프로세스 개선

**담당 조직**: CS운영팀

**무엇을**: 결제 완료 후 주문 내역 미표시 문의에 대한 즉시 확인·보상 프로세스 수립

**언제까지**: 1개월 이내

**어떻게**:
- 결제 승인 내역 조회 전용 관리자 도구 도입하여 CS 상담사가 즉시 결제 사실 확인 가능하도록 지원
- 확인 후 24시간 내 주문 수동 복구 또는 환불 처리 SLA 수립
- 유사 문의 발생 시 고객에게 선제적으로 처리 현황 안내(SMS/앱푸시) 발송

---

##### 우선순위 3 (중간) - 재발 방지 및 고객 경험 개선

**담당 조직**: 프로덕트팀, UX디자인팀

**무엇을**: 결제 완료 화면에 주문번호 즉시 표기 및 주문내역 지연 안내 문구 추가

**언제까지**: 6주 이내

**어떻게**:
- 결제 완료 시 화면에 주문번호를 즉시 생성·노출하여 결제-주문 매칭 근거 확보
- 주문 내역 반영까지 최대 소요 시간(예: "최대 5분 내 반영됩니다") 안내 문구 삽입
- 월별 결제-주문 불일치 발생 건수를 KPI로 설정하여 분기별 0.1% 이하로 감소 목표 관리

## Case 2

### Question

쿠폰을 적용했는데 결제 금액이 할인되지 않았습니다.

### Candidates

- **S0**: 쿠폰 적용 결제 금액 할인 미적용 | 근거: [VOC1] CUST092 쿠폰 적용 결제 금액 할인 미적용
- **S1**: CUST092 쿠폰 적용 결제 금액 할인 미적용 | 근거: [VOC1] CUST092 쿠폰 적용 결제 금액 할인 미적용
- **S2**: CUST092 쿠폰 적용 결제 금액 할인 미적용 | 근거: [VOC1] CUST092 쿠폰 적용 결제 금액 할인 미적용

### Policy

#### 정책 개선안: 쿠폰 할인 미적용 오류 대응

##### [우선순위 1] 긴급 시스템 점검 및 즉시 조치
**담당 조직**: 결제시스템팀, QA팀
**무엇을**: CUST092 사례와 동일한 "쿠폰 적용 후 결제 금액에 할인이 반영되지 않는" 오류의 재현 테스트 및 원인 규명
**언제까지**: 접수 후 48시간 이내
**어떻게**: 
- 결제 로그 및 쿠폰 적용 트랜잭션 데이터를 전수 조사하여 유사 사례 발생 건수 파악
- 쿠폰 검증 로직과 결제 금액 계산 로직 간 연동 오류 여부 점검
- 원인 확인 시 즉시 핫픽스 배포, 원인 미확인 시 임시 우회 로직(수동 할인 적용 프로세스) 가동

##### [우선순위 2] 피해 고객 보상 및 사후 처리
**담당 조직**: CS운영팀, 결제시스템팀
**무엇을**: CUST092를 포함한 할인 미적용 피해 고객에게 차액 환불 또는 동일 금액의 보상 쿠폰 지급
**언제까지**: 원인 확인 후 3일 이내
**어떻게**:
- 결제 데이터 기반으로 최근 30일 내 동일 오류 발생 고객 리스트 추출
- 자동 환불 프로세스 구축이 어려운 경우 CS팀이 수동으로 개별 환불 처리
- 처리 완료 후 고객에게 SMS/앱 푸시로 결과 안내

##### [우선순위 3] 재발 방지를 위한 모니터링 체계 구축
**담당 조직**: 결제시스템팀, 데이터분석팀
**무엇을**: 쿠폰 적용률 대비 실제 할인 반영률을 상시 모니터링하는 대시보드 구축
**언제까지**: 2주 이내
**어떻게**:
- "쿠폰 적용 시도 건수" 대비 "실제 할인 적용 건수" 비율을 실시간 지표로 설정
- 할인 미적용률이 0.5% 초과 시 자동 알림(Slack/이메일) 발송
- 주 1회 결제팀-QA팀 합동 리뷰 회의를 통해 이상 패턴 조기 발견

##### [우선순위 4] 재현 테스트 자동화
**담당 조직**: QA팀
**무엇을**: 쿠

## Case 3

### Question

로그인 인증번호가 너무 늦게 도착합니다.

### Candidates

- **S0**: ‘빠른 처리’를 약속했는데, 실제로는 2주 이상 지연되었습니다. | 근거: [VOC1] CUST082 사고 접수 시 ‘빠른 처리’를 약속했는데, 실제로는 2주 이상 지연되었습니다. 약속 불이행으로 신뢰가 무너졌습니다.
- **S1**: 인증번호가 늦게 와서 여러 번 시도해야 합니다. | 근거: [VOC2] CUST091 앱 로그인할 때마다 인증번호가 늦게 와서 여러 번 시도해야 합니다.
- **S2**: 배송 지연 환불 요청 답변 없음 고객 응대 지연 | 근거: [VOC4] CUST100 배송 지연 환불 요청 답변 없음 고객 응대 지연

### Policy

#### 정책 개선안: 배송 지연 환불 요청 응대 프로세스 개선

##### [우선순위 1] 환불 요청 응답 SLA 도입
- **담당 조직**: CS운영팀
- **무엇을**: 배송 지연으로 인한 환불 요청 접수 시 24시간 이내 1차 답변(접수 확인 및 처리 예정일 안내) 의무화
- **언제까지**: 2025년 8월 1일부터 즉시 시행
- **어떻게**: CS 티켓 시스템에 '배송 지연 환불' 카테고리를 신설하고, 접수 후 24시간 초과 시 자동으로 팀장에게 에스컬레이션 알림 발송

##### [우선순위 2] 미답변 건 모니터링 체계 구축
- **담당 조직**: CS품질관리팀
- **무엇을**: 환불 요청 후 48시간 이상 답변 없는 건(예: CUST100 사례)을 일간 리포트로 추출하여 관리
- **언제까지**: 2025년 8월 15일까지 대시보드 구축 완료
- **어떻게**: 티켓 시스템에서 '미답변 경과 시간' 필드를 자동 집계하고, 초과 건은 CS운영팀장에게 매일 오전 전달

##### [우선순위 3] 배송 지연 환불 처리 기준 명확화
- **담당 조직**: 물류운영팀 & CS운영팀 공동
- **무엇을**: 배송 지연 기준(예: 예정일 3일 초과)에 따른 환불 승인 기준과 처리 절차를 문서화하여 CS 상담원에게 배포
- **언제까지**: 2025년 8월 10일까지 가이드라인 완성 및 교육 실시
- **어떻게**: 상담원이 별도 승인 대기 없이 기준 충족 시 즉시 환불 처리 가능하도록 권한 위임(1차 승인권 부여)

##### [우선순위 4] 고객 재발 방지를 위한 사후 안내 강화
- **담당 조직**: CS운영팀
- **무엇을**: 환불 처리 완료 후 처리 결과와 소요 시간을 SMS/이메일로 자동 발송
- **언제까지**: 2025년 8월 20일까지 자동화 시스템 연동 완료
- **어떻게**: 환불 승인 시스템과 알림 발송 시스템(SMS/이메일) API 연동을 통해 처리 즉시 고객에게 통보

## Case 4

### Question

배송 예정일이 지났는데 배송 상태가 바뀌지 않습니다.

### Candidates

- **S0**: 배송 예정일 지연 | 근거: [VOC1] CUST093 배송 예정일 지연 배송 상태 미변경 배송 조회
- **S1**: 배송 상태 미변경 | 근거: [VOC1] CUST093 배송 예정일 지연 배송 상태 미변경 배송 조회
- **S2**: 배송 조회 | 근거: [VOC1] CUST093 배송 예정일 지연 배송 상태 미변경 배송 조회

### Policy

#### 배송 조회 서비스 개선안

##### 우선순위 1 (긴급) — 배송 상태 실시간 업데이트 시스템 구축
- **담당 조직**: 물류운영팀, IT개발팀
- **무엇을**: 배송 상태(집화-이동중-배송지도착-배송완료)가 실제 상황과 불일치하는 문제 해결을 위해 택배사 API 연동 주기를 기존 1일 1회에서 **최소 4시간 단위 자동 갱신**으로 변경
- **언제까지**: 2025년 4분기 내 시스템 반영 완료
- **어떻게**: 택배사 실시간 배송 데이터 API 연동, 배송상태 미변경 12시간 초과 시 시스템 자동 알림 및 CS팀 에스컬레이션 프로세스 구축

##### 우선순위 2 (중요) — 배송 지연 사전 고지 프로세스 도입
- **담당 조직**: CS운영팀, 물류운영팀
- **무엇을**: 배송 예정일 초과가 예상되는 건에 대해 **고객에게 사전 SMS/앱 알림 발송** (지연 사유, 변경된 예정일 포함)
- **언제까지**: 2025년 3분기 내 시범 운영, 4분기 정식 적용
- **어떻게**: 배송 예정일 D-1 기준 배송 상태가 '이동중' 이전 단계일 경우 자동으로 지연 예상 알림 발송 트리거 설정

##### 우선순위 3 (개선) — 고객 배송 조회 화면 고도화
- **담당 조직**: UX디자인팀, IT개발팀
- **무엇을**: 배송 조회 페이지에 **최종 업데이트 시각 명시** 및 "배송 지연 문의" 버튼 신설로 CS 문의 채널 즉시 연결
- **언제까지**: 2025년 3분기 내 UI 반영
- **어떻게**: 배송 조회 화면 내 상태 변경 이력(타임스탬프 포함) 표시, 문의 버튼 클릭 시 해당 주문번호 자동 연동된 CS 채팅 상담 연결

##### 효과 측정 지표(KPI)
- 배송 상태 미변경 관련 VOC 접수 건수: **월 30% 이상 감소** 목표
- 배송 지연 사전 고지 발송률: **95% 이상** 달성
- 배송 조회 관련 고객 만족도(CSAT): **분기별 5%p 상승**

## Case 5

### Question

환불 신청을 했는데 처리 상태를 알 수 없습니다.

### Candidates

- **S0**: CUST054 청구 진행 상황을 확인하려고 앱을 열었는데, 진행 현황이 제대로 표시되지 않았습니다. 투명한 프로세스 안내가 부족합니다. | 근거: [VOC1] CUST054 청구 진행 상황을 확인하려고 앱을 열었는데, 진행 현황이 제대로 표시되지 않았습니다. 투명한 프로세스 안내가 부족합니다.
- **S1**: CUST094 환불 신청 처리 상태 상태 확인 불가 | 근거: [VOC2] CUST094 환불 신청 처리 상태 상태 확인 불가
- **S2**: CUST054 청구 진행 상황을 확인하려고 앱을 열었는데, 진행 현황이 제대로 표시되지 않았습니다. 투명한 프로세스 안내가 부족합니다. | 근거: [VOC1] CUST054 청구 진행 상황을 확인하려고 앱을 열었는데, 진행 현황이 제대로 표시되지 않았습니다. 투명한 프로세스 안내가 부족합니다.

### Policy

#### 정책 개선안: 청구 진행 현황 표시 개선

##### 우선순위 1 (긴급, 2주 이내 착수)

**담당 조직:** IT개발팀 · 앱UX팀

**무엇을:** 청구 진행 상황 실시간 표시 오류 긴급 점검 및 수정
- 현재 앱 내 청구 진행 현황이 표시되지 않는 버그의 원인 파악(서버 연동 오류, 캐싱 문제, UI 렌더링 오류 등)
- 청구 접수 → 심사중 → 승인/보완요청 → 지급완료의 4단계 진행 바(progress bar) 구현

**언제까지:** 2025년 11월 말까지 긴급 패치 배포

**어떻게:**
1. 1주차: 장애 로그 분석 및 원인 규명
2. 2주차: 수정사항 QA 테스트 후 앱 업데이트 배포
3. 배포 후 1주간 CUST054 포함 유사 민원 발생 고객 대상 정상 작동 여부 재확인 콜 실시

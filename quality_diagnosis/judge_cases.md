## Case 1

### Question

결제는 완료되었는데 주문 내역에 보이지 않습니다.

### Candidates

- **S0**: 결제는 완료되었는데 주문 내역에 보이지 않습니다. | 근거: [VOC1] CUST090 결제는 완료되었는데 주문 내역에 보이지 않습니다.
- **S1**: 결제 완료 주문 내역 미노출 주문 누락 | 근거: [VOC2] CUST103 결제 완료 주문 내역 미노출 주문 누락
- **S2**: 결제는 완료되었는데 주문 내역에 보이지 않습니다. | 근거: [VOC1] CUST090 결제는 완료되었는데 주문 내역에 보이지 않습니다.

### Policy

#### 정책 개선안: 결제-주문 내역 동기화 오류 개선

##### 우선순위 1 (긴급, 2주 이내)
**담당 조직**: IT개발팀(백엔드/결제 시스템 파트)
**무엇을**: 결제 완료 트랜잭션과 주문 내역 DB 간 실시간 동기화 로직 점검 및 오류 수정
**언제까지**: 착수 후 2주 이내 원인 진단 완료, 4주 이내 수정 배포
**어떻게**:
- 결제 승인(PG사 콜백) 시점과 주문 생성(Order Create) 시점 간 API 호출 실패 로그 전수 조사
- 결제 완료 후 주문 미생성 건에 대한 자동 재시도(Retry) 로직 구축 (최대 3회, 5분 간격)
- 동기화 실패 시 즉시 알림 발송하는 모니터링 대시보드 구축(Slack/이메일 연동)

---

##### 우선순위 2 (단기, 1개월 이내)
**담당 조직**: CS운영팀
**무엇을**: 결제 완료 후 주문 미확인 고객 대상 임시 대응 프로세스 수립
**언제까지**: 1개월 이내 시행
**어떻게**:
- 결제 완료 시각 기준 10분 내 주문 내역 미반영 시 고객에게 선제적 안내 문자/알림톡 자동 발송
- CS 상담원용 "결제-주문 불일치 조회 툴" 도입하여 즉시 주문 상태 확인 및 수동 처리 가능하도록 지원
- 해당 문의 유형 발생 시 24시간 이내 1차 답변, 48시간 이내 완전 해결을 SLA로 설정

---

##### 우선순위 3 (재발 방지, 분기별)
**담당 조직**: QA팀 + IT개발팀
**무엇을**: 결제-주문 연동 프로세스에 대한 정기 부하 테스트 및 장애 시나리오 테스트 실시
**언제까지**: 분기 1회(3개월 주기) 정기 시행
**어떻게**:
- 결제 폭주 상황(프로모션/세일 기간) 가정한 스트레스 테스트 수행
- 결제-주문 불일치 발생률을 핵심 지표(KPI)로 설정하여 월별 0.1% 미만 목표 관리
- 테스트 결과 기반 시스템 개선 사항을 분기별 보고서로 경영진 공유

## Case 2

### Question

쿠폰을 적용했는데 결제 금액이 할인되지 않았습니다.

### Candidates

- **S0**: 쿠폰 적용 결제 금액 할인 미적용 | 근거: [VOC1] CUST092 쿠폰 적용 결제 금액 할인 미적용
- **S1**: 결제 오류 쿠폰 소멸 | 근거: [VOC2] CUST099 결제 오류 쿠폰 소멸 쿠폰 사라짐
- **S2**: 쿠폰 사라짐 | 근거: [VOC2] CUST099 결제 오류 쿠폰 소멸 쿠폰 사라짐

### Policy

#### 정책 개선안: 결제 오류 시 쿠폰 소멸 문제 개선

##### 【우선순위 1】 결제 오류 발생 시 쿠폰 자동 복구 시스템 구축
- **담당 조직**: 결제개발팀, 쿠폰/프로모션팀
- **무엇을**: 결제 실패(오류) 발생 시 사용 처리된 쿠폰을 자동으로 원상 복구(재발급 또는 상태 롤백)하는 로직 구축
- **언제까지**: 2주 내 원인 분석 완료, 4주 내 시스템 반영 및 배포
- **어떻게**: 결제 트랜잭션과 쿠폰 사용 처리를 하나의 트랜잭션으로 묶어(원자적 처리) 결제 실패 시 쿠폰 상태도 함께 롤백되도록 구현. 배포 후 2주간 QA 및 실사용 모니터링 진행

##### 【우선순위 2】 쿠폰 소멸 이력 조회 및 고객 셀프 복구 기능 제공
- **담당 조직**: CS운영팀, 프론트엔드개발팀
- **무엇을**: 고객이 마이페이지에서 쿠폰 사용/소멸 이력을 직접 확인하고, 결제 오류로 인한 소멸 건은 자동 재발급 신청 가능하도록 기능 추가
- **언제까지**: 6주 내 기능 오픈
- **어떻게**: 결제 실패 로그와 쿠폰 소멸 로그를 매칭하는 백엔드 API 개발 후, 프론트엔드에 '쿠폰 복구 신청' 버튼 추가. CS팀은 신청 건에 대해 24시간 이내 처리 완료

##### 【우선순위 3】 재발 방지를 위한 실시간 모니터링 및 알림 체계 마련
- **담당 조직**: 인프라/모니터링팀, 쿠폰/프로모션팀
- **무엇을**: 결제 오류로 인한 쿠폰 소멸 건수를 실시간 집계하는 대시보드 구축 및 임계치 초과 시 자동 알림 발송
- **언제까지**: 8주 내 대시보드 구축 완료
- **어떻게**: 결제 오류 로그와 쿠폰 상태 변경 로그를 연동한 모니터링 대시보드 개발, 일 평균 소멸

## Case 3

### Question

로그인 인증번호가 너무 늦게 도착합니다.

### Candidates

- **S0**: ‘빠른 처리’를 약속했는데, 실제로는 2주 이상 지연되었습니다. | 근거: [VOC1] CUST082 사고 접수 시 ‘빠른 처리’를 약속했는데, 실제로는 2주 이상 지연되었습니다. 약속 불이행으로 신뢰가 무너졌습니다.
- **S1**: 인증번호가 늦게 와서 여러 번 시도해야 합니다. | 근거: [VOC2] CUST091 앱 로그인할 때마다 인증번호가 늦게 와서 여러 번 시도해야 합니다.
- **S2**: 배송 예정일 지연 배송 상태 미변경 배송 조회 | 근거: [VOC3] CUST093 배송 예정일 지연 배송 상태 미변경 배송 조회

### Policy

#### 정책 개선안: 배송 지연 환불 요청 고객 응대 프로세스 개선

##### [우선순위 1] 환불 요청 응대 SLA(서비스 수준 협약) 신설
- **담당 조직**: CS운영팀
- **무엇을**: 배송 지연으로 인한 환불 요청 건에 대해 최초 응답 시간을 24시간 이내, 처리 완료까지 72시간 이내로 규정하는 응대 SLA 도입
- **언제까지**: 2025년 8월 말까지 SLA 기준 수립 및 시행
- **어떻게**: CS 티켓 시스템에 '배송지연 환불' 전용 카테고리를 신설하고, 접수 즉시 담당자 자동 배정 및 24시간 초과 시 관리자에게 에스컬레이션 알림이 발송되도록 시스템 구축

##### [우선순위 2] 무응답 건 모니터링 대시보드 구축
- **담당 조직**: CS운영팀 + IT개발팀
- **무엇을**: 환불 요청 후 답변 없이 방치된 건(48시간 초과)을 실시간으로 추적하는 대시보드 개발
- **언제까지**: 2025년 9월 말까지 1차 버전 오픈
- **어떻게**: CRM 데이터와 연동하여 미응답 건수, 평균 응답 소요시간, 담당자별 처리 현황을 일간 리포트로 자동 생성 및 팀장에게 공유

##### [우선순위 3] 배송 지연 사전 고지 및 자동 환불 안내 프로세스 도입
- **담당 조직**: 물류팀 + CS운영팀
- **무엇을**: 배송 예정일 초과 시 고객에게 지연 사유와 환불/재배송 옵션을 자동 안내하는 메시지 발송 체계 구축
- **언제까지**: 2025년 10월 초까지 시범 운영 시작
- **어떻게**: 물류 시스템의 배송 상태 데이터를 CS 시스템과 연동하여, 예정일 초과 즉시 SMS/앱 푸시로 안내 문구 자동 발송 (환불 신청 링크 포함)

##### [우선순위 4] 고객 응대 지연 재발 방지를 위한 인력 운영 개선
- **담당 조직**: CS운영팀 + 인사팀
- **무엇을**: 환불 문의 급증 시기(성수기, 프로모션 직후)

## Case 5

### Question

환불 신청을 했는데 처리 상태를 알 수 없습니다.

### Candidates

- **S0**: 환불 신청 처리 상태 상태 확인 불가 | 근거: [VOC1] CUST094 환불 신청 처리 상태 상태 확인 불가
- **S1**: CUST094 환불 신청 처리 상태 상태 확인 불가 | 근거: [VOC1] CUST094 환불 신청 처리 상태 상태 확인 불가
- **S2**: CUST094 환불 신청 처리 상태 상태 확인 불가 | 근거: [VOC1] CUST094 환불 신청 처리 상태 상태 확인 불가

### Policy

#### 정책 개선안: 환불 신청 처리 상태 조회 시스템 개선

##### [우선순위 1] 환불 처리 상태 실시간 조회 기능 구축
- **담당 조직**: IT개발팀, CS운영팀
- **무엇을**: 고객이 마이페이지 또는 앱에서 환불 신청 진행 상태(접수완료/심사중/승인/이체완료)를 실시간으로 조회할 수 있는 기능 개발
- **언제까지**: 2025년 12월 31일까지 (개발 완료 및 오픈)
- **어떻게**: 환불 처리 시스템(내부 ERP)과 고객 조회 화면(웹/앱)을 API로 연동하여 상태값을 자동 동기화, 상태 변경 시 실시간 반영

##### [우선순위 2] 환불 처리 단계별 자동 알림(SMS/앱푸시) 발송
- **담당 조직**: CS운영팀, 마케팅기획팀
- **무엇을**: 환불 신청 접수, 심사 완료, 입금 완료 등 주요 단계마다 고객에게 자동 알림 발송
- **언제까지**: 2025년 11월 30일까지 (알림 발송 체계 구축)
- **어떻게**: 기존 알림 발송 시스템(SMS/앱푸시 API)에 환불 상태 변경 이벤트를 트리거로 연동, 발송 성공률 95% 이상 확보

##### [우선순위 3] CS 상담 시 환불 상태 즉시 확인 매뉴얼 마련
- **담당 조직**: CS교육팀, CS운영팀
